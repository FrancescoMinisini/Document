\subsection{Compared methods}
Within this common framework, four optimization and comparison schemes are considered. 
\begin{itemize}
    \item The first is a direct gradient-based baseline, implemented through the Adam optimizer, which searches directly over the filtered control trajectory for a fixed gate horizon. 
    \item The second is a policy-gradient reinforcement-learning controller trained in the nominal environment, that is, without stochastic perturbations applied during training. 
    \item The third is the noise-optimized TRPO controller, which is trained in a stochastic environment where Gaussian perturbations are injected into the control amplitudes and into the anharmonicity-related parameter entering the Hamiltonian. 
    \item The fourth comparison is not a pulse-level optimizer but a circuit-level reference, namely the optimal gate-synthesis baseline used to benchmark runtime on the family \(N(\alpha,\alpha,\gamma)\). 
\end{itemize}
These four methods play different methodological roles. The Adam baseline tests whether a standard first-order optimization of the raw control variables is sufficient to obtain high-quality pulses. The nominal TRPO controller tests whether reinforcement learning is already competitive in the idealized deterministic setting. The noise-optimized TRPO controller tests whether robustness can be learned directly from stochastic training. Finally, the gate-synthesis baseline tests whether direct analog control offers a runtime advantage over a discrete compiled implementation of the same target gate. Taken together, these methods define a comparison protocol that is not centered on a single optimizer, but on a set of scientifically distinct questions concerning accuracy, robustness, and hardware efficiency.

\subsection{Metrics}
The performance of each control method is evaluated through a set of complementary metrics designed to probe different aspects of the control problem. At the most basic level, the implemented gate is assessed through its fidelity with respect to the target unitary. For a final physical propagator \(U(T)\), projected onto the computational subspace, the gate fidelity is computed as
\begin{equation}
F_{\mathrm{gate}}
=
\frac{1}{16}
\left|
\mathrm{Tr}\!\left(
U_{\mathrm{proj}}^{\dagger}(T)\,U_{\mathrm{target}}
\right)
\right|^2,
\label{eq:validation_gate_fidelity}
\end{equation}
where \(U_{\mathrm{target}}=N(\alpha,\alpha,\gamma)\). This quantity is reported together with the total UFO cost, which combines fidelity, leakage, boundary penalties, and runtime into a single scalar objective. Since the framework is explicitly multilevel, nominal performance is not characterized by fidelity alone. For every optimized control plan, the simulation also returns the TSWT-based leakage estimate \(L_{\mathrm{tot}}\), its decomposition into boundary and integral contributions, the boundary penalty term, and the final runtime in nanoseconds. These quantities allow one to distinguish between a control that is merely accurate on the logical subspace and one that is also physically well behaved in the full truncated Hilbert space. In addition to nominal performance, robustness is evaluated statistically under sampled noisy control realizations. Given a control plan \(\tilde{\bm{c}}\) and a Gaussian noise level \(\sigma_{\mathrm{noise}}\), the key robustness diagnostics are the average fidelity
\begin{equation}
\overline{F}
=
\mathbb{E}_{\delta\tilde{\bm{c}}}
\Bigl[
F\bigl(U(\tilde{\bm{c}}+\delta\tilde{\bm{c}})\bigr)
\Bigr],
\label{eq:validation_average_fidelity}
\end{equation}
the corresponding average gate fidelity, and the fidelity variance
\begin{equation}
\mathrm{Var}(F)
=
\mathbb{E}_{\delta\tilde{\bm{c}}}
\left[
\bigl(F-\overline{F}\bigr)^2
\right].
\label{eq:validation_fidelity_variance}
\end{equation}
These metrics are essential because they distinguish mean robustness from shot-to-shot stability: a control scheme with high average fidelity but large variance remains experimentally fragile. Finally, for the runtime study, the relevant metric is the total control duration required to realize a target gate, compared against the optimal discrete gate-synthesis baseline. The resulting evaluation protocol is therefore explicitly multidimensional: it measures logical accuracy, leakage suppression, robustness under stochastic perturbations, and hardware efficiency rather than relying on a single scalar score.

\subsection{Evaluation protocol}
All experiments follow a common procedural logic. For a given target gate \(N(\alpha,\alpha,\gamma)\), a control method is first trained or optimized under its own learning rule until either a prescribed iteration budget is exhausted or a termination criterion on the UFO cost is satisfied. The resulting best control plan is then stored and re-evaluated in a deterministic nominal setting in order to extract its final cost, fidelity, leakage, and runtime under identical physical assumptions. For robustness studies, the same saved control plan is subsequently subjected to repeated noisy rollouts, in which Gaussian perturbations are independently sampled and injected into the control channels and into the anharmonicity-related parameter of the Hamiltonian. This Monte Carlo evaluation yields the empirical estimates entering Eqs.~\eqref{eq:validation_average_fidelity} and \eqref{eq:validation_fidelity_variance}. In the runtime study, by contrast, the focus is not on statistical robustness but on how the optimized gate duration varies across the target family, especially along the subfamily \(N(\alpha,\alpha,\pi/2)\), where analog control can be meaningfully compared with optimal gate synthesis. The same timestep, control bounds, bandwidth assumptions, and physical truncation are retained throughout, so that differences between methods can be attributed to the optimization strategy rather than to inconsistencies in the simulation model. This common evaluation protocol is particularly important for the interpretation of the results: it ensures that the subsequent comparisons are methodologically fair, while also making the distinction between nominal accuracy, noisy robustness, and runtime efficiency completely explicit.

\subsection{Parameter selection and practical trade-offs}
\label{subsec:parameter_selection_tradeoffs}

A separate discussion of parameter selection is warranted because, in reinforcement-learning-based quantum control, the numerical setup is not merely an implementation detail but part of the methodology itself. One of the main practical limitations of reinforcement learning is its strong sensitivity to hyperparameter choice, including optimizer settings, exploration scale, rollout horizon, and policy architecture. This issue is compounded by the large dimensionality of the hyperparameter space, which makes a truly exhaustive search computationally expensive and, in many cases, infeasible. For this reason, the parameter choices adopted in this work are not presented as globally optimal in any absolute sense. Rather, they are chosen through a limited but physically informed sensitivity study aimed at identifying settings that are stable, interpretable, and sufficiently effective for the numerical validation of the framework. This point is important for the interpretation of the results: the experiments below should be read as validating the method under a coherent and justified parameter regime, not as claiming that every reported number is the outcome of a provably optimal hyperparameter configuration.

In order to keep the parameter study computationally manageable, tuning was performed on a representative target gate taken from the family \(N(\alpha,\alpha,\gamma)\), rather than over the full continuous target manifold. This choice is methodologically motivated. As discussed in Section~\ref{subsec:target_gate_family}, the family \(N(\alpha,\alpha,\gamma)\) is broad enough to contain physically and algorithmically relevant two-qubit gates, but sufficiently structured that a single representative point can serve as a useful proxy for hyperparameter selection. The idea is therefore to calibrate the learning dynamics on one nontrivial target and then reuse the resulting settings across the broader validation study. This avoids conflating two distinct sources of variation: changes in optimization behavior caused by hyperparameters, and changes in control difficulty caused by the target gate itself. In this sense, the representative target is not introduced because it is uniquely special, but because it provides a stable reference point from which the rest of the experimental study can be organized.

Among the TRPO-specific hyperparameters, two are particularly important: the initial policy dispersion, controlled by \texttt{init\_log\_std}, and the trust-region radius, controlled by \texttt{max\_kl}. The parameter \texttt{init\_log\_std} determines the initial exploration scale of the stochastic policy and therefore affects how broadly the agent samples the continuous control space during the early stages of training. If the initial exploration is too small, the policy may become trapped in a narrow and suboptimal region of the control landscape; if it is too large, the sampled trajectories may become excessively noisy and learning may become unstable. The parameter \texttt{max\_kl}, by contrast, controls the maximum allowed Kullback--Leibler divergence between two consecutive policies in TRPO. It therefore governs a trade-off between speed of improvement and update reliability: larger values permit faster policy movement but increase the risk of overshooting into poor regions of policy space, whereas smaller values produce more conservative and stable updates at the cost of slower progress. In a continuous-control problem such as the present one, where the reward depends on a long pulse trajectory and the environment may be stochastic, both parameters have a direct effect on whether training is exploratory enough to discover good controls while remaining stable enough to refine them.

A second major set of design choices concerns the temporal discretization of the control problem. The timestep \(\Delta t\) determines the temporal mesh on which the controls are defined, while the maximum duration \(T_{\max}\) determines the largest admissible episode length. These choices involve a clear trade-off. A smaller timestep provides a finer control mesh and therefore a more expressive pulse representation, but it also increases the dimensionality of the action trajectory, the computational cost of simulation, and the difficulty of optimization. A larger timestep reduces computational burden, but may artificially restrict the control space and make it harder to resolve fast but physically meaningful pulse features. Similarly, a longer maximum duration allows the optimizer to discover slower but potentially easier solutions, whereas a shorter horizon encourages time-efficient control but increases the difficulty of the search. The selected values of \(\Delta t\) and \(T_{\max}\) should therefore be understood as compromise choices: they are fine enough to represent nontrivial pulse dynamics and allow meaningful comparison with the reference framework, but coarse enough to keep the experiments computationally tractable.

The remaining parameter choices are tied more directly to the physical modeling assumptions of the framework. The weights of the UFO cost function, \((\chi,\beta,\mu,\kappa)\), determine the relative importance assigned to gate accuracy, leakage suppression, boundary regularity, and runtime reduction, and therefore encode an explicit multi-objective trade-off rather than a purely numerical preference. Likewise, the amplitude and structure of the training noise are chosen in direct reference to the stochastic control model used in the original paper, so that the robustness experiments remain comparable to the intended reference setting rather than introducing an unrelated perturbation model.
